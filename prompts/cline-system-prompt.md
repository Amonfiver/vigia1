Trabajamos en modo SDD.

Reglas generales:
1. Antes de hacer cambios grandes, leer y respetar la spec del proyecto.
2. No inventar funcionalidades fuera del alcance actual.
3. Hacer cambios pequeños, controlados y compilables.
4. Priorizar claridad, arquitectura extensible y estabilidad.
5. Mantener el foco en el MVP actual.
6. No intentar resolver demasiadas cosas en una sola iteración.

Regla obligatoria de documentación por archivo:
Cada archivo de código creado o modificado debe incluir al inicio una cabecera breve de documentación explicando:
- propósito del archivo,
- responsabilidad principal,
- alcance dentro del sistema,
- decisiones técnicas relevantes,
- limitaciones temporales del MVP,
- cambios recientes si aplica.

La cabecera debe estar escrita para que un futuro programador pueda abrir el archivo y entender rápidamente de qué trata sin tener que deducirlo todo por contexto.

Al terminar cada tarea:
- explica qué hiciste,
- lista los archivos creados o modificados,
- indica riesgos o limitaciones,
- recomienda el siguiente paso.

## Documentación viva obligatoria

Este proyecto mantiene documentación viva para permitir continuidad entre agentes y desarrolladores.

Antes de cada tarea, debes leer:
- docs/project-status.md
- docs/dev-log.md
- docs/code-map.md

Después de cada tarea con cambios relevantes, debes actualizar:
- docs/project-status.md
- docs/dev-log.md
- docs/code-map.md

La documentación debe reflejar el estado real del proyecto, sin exagerar avances ni ocultar limitaciones temporales.
